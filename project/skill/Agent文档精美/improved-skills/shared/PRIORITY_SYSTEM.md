# Priority System (P0-P3) — 规则优先级分层

## 设计哲学

规则过多时，模型会选择性失忆。必须分层，让Agent知道：
- **哪些是绝对红线**（不满足就不能交付）
- **哪些是用户契约**（不满足就是违约）
- **哪些是质量标准**（尽量满足，失败要说明原因）
- **哪些是风格偏好**（可根据任务灵活调整）

---

## P0 Safety / Validity Gate — 安全门

> **不满足就绝对不能交付。这是生死线。**

### 通用P0规则（所有格式）

| # | 规则 | 检查方式 |
|---|------|----------|
| 1 | 文件能正常打开，不被目标应用程序拒绝 | 用对应工具打开验证 |
| 2 | 无结构性错误（schema validation通过） | 专用验证工具 |
| 3 | 无运行时错误（公式不报错、链接不损坏） | recheck/validate |
| 4 | 不编造数据、不伪造引用 | 搜索验证 |
| 5 | 不包含占位符文本 | 文本扫描 |

### 格式特定P0

**DOCX:**
- RunProperties中RunFonts必须是第一个子元素
- 表格GridColumn宽度与单元格宽度匹配
- 图片正确嵌入（build输出显示images > 0）

**XLSX:**
- 无公式错误（#VALUE!, #DIV/0!, #REF!）
- 无隐式数组公式（MS Excel兼容）
- 不使用XLOOKUP/FILTER/UNIQUE等不兼容函数

**PDF:**
- HTML无CSS Counter（使用data-*属性替代）
- Mermaid图表≤8节点
- 分页正确（无内容截断）

**PPTX:**
- check.sh报告0 errors
- 无文字溢出/遮挡/漂移
- 主题$key引用全部解析正确

## P1 User Contract — 用户契约

> **用户明确要求的内容，必须满足。**

| # | 规则 | 示例 |
|---|------|------|
| 1 | 用户要求的格式必须输出 | "我要PPT" → 必须输出.pptd |
| 2 | 用户要求的内容必须包含 | "要有财务预测" → 必须有财务预测页 |
| 3 | 用户指定的风格/配色必须遵循 | "用公司品牌色" → 使用品牌色 |
| 4 | 用户提供的素材必须使用 | "参考这份PDF" → 必须读取并参考 |
| 5 | 用户要求的字数/页数必须达标 | "3000字" → 2400-3600字 |

### 冲突解决

当P1规则之间冲突时，按以下优先级：
```
格式要求 > 内容要求 > 风格要求 > 数量要求
```

示例：用户要求"输出PDF"（格式）和"要有可编辑的表格"（内容）冲突 → 
PDF格式优先，表格用三线表实现（非可编辑，但最符合PDF能力）。

## P2 Quality Standard — 质量标准

> **尽量满足，失败要说明原因。**

| # | 规则 | 重要性 |
|---|------|--------|
| 1 | 封面有视觉设计（非纯文本） | 高 |
| 2 | 配色使用预设方案（非即兴选择） | 高 |
| 3 | 图表有标题和轴标签 | 高 |
| 4 | 外部数据有来源引用 | 高 |
| 5 | 页眉页脚完整 | 中 |
| 6 | 目录可自动更新 | 中 |
| 7 | 交替行颜色增强可读性 | 中 |
| 8 | 条件格式增强数据可读性 | 低 |

### 豁免条件

以下情况可豁免P2规则，但必须在交付说明中注明：
- 用户明确要求"简单/快速" → 可简化封面
- 数据来源为用户自有数据（无外部引用需求） → 可省略来源列
- 时间紧迫的草稿版本 → 可省略条件格式

---

## P3 Style Preference — 风格偏好

> **可根据任务灵活调整。这些不是规则，是建议。**

| # | 建议 | 默认行为 |
|---|------|----------|
| 1 | 使用低饱和度配色 | 默认遵循，用户可覆盖 |
| 2 | 正文不低于18px（PPT） | 默认遵循，高密度场景可降至16px |
| 3 | 一页一个核心信息（PPT） | 默认遵循，复杂数据页可例外 |
| 4 | 使用三线表（学术风格） | 默认遵循，商务场景可改用深色表头 |
| 5 | 封面使用Hero/Magazine/Swiss风格 | 默认Swiss，用户可指定 |

## 优先级执行流程

```
交付前检查：
  P0: ALL PASS → 继续
      ANY FAIL → 禁止交付，修复后再检查
  
  P1: ALL PASS → 继续
      ANY FAIL → 警告，修复或向用户说明
  
  P2: 统计通过数/总数
      ≥80% PASS → 可交付，注明未通过项
      <80% PASS → 建议修复后再交付
  
  P3: 统计通过数/总数
      不阻塞交付
      未通过项作为改进建议列出
```

## 规则冲突示例与解决方案

### 示例1: "不要额外文件" vs "输出review.md"

```
规则A: "不得提供用户没要求的文件"（P2）
规则B: "分析流程要求输出review.md"（P2）

冲突解决:
- review.md是工作流中间产物，不是交付物
- 最终交付只包含用户要求的.xlsx
- review.md用于自检，不随文件一起交付
```

### 示例2: "必须隐藏网格线" vs "用户要求能看到网格"

```
规则A: "所有工作表必须隐藏网格线"（P3默认）
规则B: "用户要求能看到网格"（P1）

冲突解决:
- P1 > P3
- 显示网格线，遵循用户要求
```

### 示例3: "正文不低于18px" vs "一页要放大量数据"

```
规则A: "正文不低于18px"（P3）
规则B: "一页要放大量数据"（P1内容要求）

冲突解决:
- 降至16px（P3可调整），同时优化布局减少文字量
- 如果仍放不下，向用户建议分页或精简内容
```
